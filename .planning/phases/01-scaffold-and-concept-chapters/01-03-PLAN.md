src/ch02-openhands/actions-observations.md (keep H1 `# 액션과 관찰 타입`):
- 핵심 사이클: Agent가 ActionEvent emit → Workspace 실행 → ObservationEvent를 EventLog에 append (ARCHITECTURE.md 2.4-2.5, 01-RESEARCH.md 사실 #3).
- 이 튜토리얼에 등장할 주요 action 타입을 표 또는 불릿으로: CmdRunAction(shell, → CmdOutputObservation: stdout/stderr/exit code), FileWriteAction(파일 생성), FileEditAction(str_replace 편집, → diff), FileReadAction, AgentThinkAction(부수효과 없는 추론), AgentFinishAction(루프 종료).
- ObservationEvent가 담는 것: stdout+stderr+exit code / diff / content. 자가 수정(self-correction)이 별도 기능이 아니라 관찰 루프의 부수효과임을 설명: 실패한 명령의 stderr/exit code가 EventLog에 들어가 다음 프롬프트에서 LLM이 보고 수정 action을 emit한다(ARCHITECTURE.md 2.6).
- 매핑 명시: ActionEvent + LLM에 전달되는 tool schema = 1부의 'tool calling'; ObservationEvent = 'observation'. 어휘 장 링크 허용: `[핵심 개념과 용어](../ch01-agentic-ai/concepts.md)`.
- ~40-60 lines.

src/ch02-openhands/runtime.md (keep H1 `# 런타임과 샌드박스`):
- Workspace 추상화: agent 코드는 workspace와 무관하게 동일; workspace는 생성 시 주입(ARCHITECTURE.md 2.7). 세 종류 표: LocalWorkspace(호스트 프로세스), DockerWorkspace(컨테이너+내부 서버, 이 튜토리얼이 사용), RemoteAPIWorkspace(네트워크 RPC).
- DockerWorkspace 내부: 컨테이너 안에서 FastAPI Action Execution Server가 `POST /execute_action`을 받음; tmux 기반 지속 bash 세션(cd 상태 유지); 지속 IPython 커널. action은 JSON POST, observation은 JSON 응답.
- 매핑 명시: DockerWorkspace + Action Execution Server = 1부의 'sandbox / isolation' 개념. (V1 용어 'workspace'를 사용하고, V0의 monolithic 'sandbox'와 혼동하지 말 것 — 격리 환경이라는 의미로만 '샌드박스' 사용.)
