<design_constraints>
These are NON-NEGOTIABLE. They come from 03-CONTEXT.md (locked decisions) and 03-RESEARCH.md (prescriptive findings):

1. **Fresh-conversation reality:** Each `openhands --headless -t "..."` is a brand-new conversation with ZERO memory of prior invocations. Files written by a prior task persist on the host filesystem, but the agent only knows about them if the prompt says so. Therefore every prompt MUST open by stating the working directory and describing which files already exist (and that the agent should `ls`/`cat` to inspect them).

2. **Workdir layout:** OPENHANDS_WORK_DIR = `/Users/ohama/projs/OpenHandsTests/oh-workdir`. The agent's initial cwd IS that dir. The calculator project lives in the `calc/` subdirectory. Every prompt must instruct the agent to work inside `calc/` (e.g. "the project is in the calc/ subdirectory; cd into it"). Task 1 creates `calc/`.

3. **Goal + constraints prompt style:** State the goal and the hard constraints (use FsLex/FsYacc, integers only, CLI-arg invocation `calc "<expr>"`). Let the agent decide HOW (grammar shape, AST). Do NOT write the lexer/parser/evaluator code for the agent EXCEPT the scaffold's .fsproj (see #4).

4. **Scaffold hands over the known-good .fsproj (research A2) VERBATIM.** This is the one piece of build scaffolding we provide, because the `# 0 ""` line-directive incompatibility (FsLexYacc 11.3.0 + .NET 10, research A3) is an obscure ENVIRONMENT quirk a 35B model is unlikely to self-fix; leaving it out would stall the run on a non-instructive bug. Pinning build scaffolding is implementation detail, not cheating. The embedded .fsproj must include: OutputType Exe + net10.0; the `<FsYacc Include="Parser.fsy"><OtherFlags>--module Parser</OtherFlags></FsYacc>` and `<FsLex Include="Lexer.fsl"><OtherFlags>--unicode --module Lexer</OtherFlags></FsLex>` item groups; the `FixLineDirectives` target with `BeforeTargets="CoreCompile" DependsOnTargets="CallFsYacc;CallFsLex"` running the two `sed -i '' '/^# 0/d'` commands; the Compile order Parser.fsi -> Parser.fs -> Lexer.fs -> Program.fs; and `<PackageReference Include="FsLexYacc" Version="11.3.0" />`. Copy it exactly from 03-RESEARCH.md section A2.

5. **NEVER reveal the %left ordering.** No prompt may mention `%left`, precedence declaration ordering, or associativity as a hint. The parser prompt (task 3) asks for a grammar over `+ - * /` and parens with correct precedence as a GOAL, but must not tell the agent how to declare it. This lets the naive-grammar associativity bug surface honestly.

6. **The build-and-test prompt (task 5) MUST require all THREE cases:** `2+3*4` -> `14`, `(2+3)*4` -> `20`, and `10-3-2` -> `5`. Per research A5, the first two pass even with the buggy naive grammar; only `10-3-2=5` exposes the missing/incorrect `%left` (naive grammar yields `10-3-2=9`). Without case 3 the bug is invisible and RUN-03 cannot be satisfied honestly.

7. **task6-fix.txt is CONDITIONAL** — used by plan 03-02 only if task 5 surfaces a genuine failure (wrong output or build error). Follow the research B6 fix-prompt pattern: state the workdir, that the FsLex/FsYacc project exists and builds but gives wrong results, show the FAILING case with its wrong actual output as a placeholder to be filled in at run time (e.g. `10-3-2 prints <ACTUAL> (should be 5)`), list all three required passing outputs, and restrict scope to Parser.fsy / "do not rewrite from scratch". It must still NOT name `%left`.
</design_constraints>
