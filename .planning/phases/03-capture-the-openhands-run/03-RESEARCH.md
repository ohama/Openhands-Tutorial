### A2. Known-Good .fsproj Skeleton (VERIFIED WORKING ON NET10 + FSLEXYACC 11.3.0)

```xml
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <OutputType>Exe</OutputType>
    <TargetFramework>net10.0</TargetFramework>
  </PropertyGroup>

  <ItemGroup>
    <FsYacc Include="Parser.fsy">
      <OtherFlags>--module Parser</OtherFlags>
    </FsYacc>
    <FsLex Include="Lexer.fsl">
      <OtherFlags>--unicode --module Lexer</OtherFlags>
    </FsLex>
  </ItemGroup>

  <!-- REQUIRED WORKAROUND for .NET 10 + FsLexYacc 11.3.0:
       fsyacc generates "# 0 """" line directives that F# 10 compiler rejects.
       Strip them with a post-generation sed step. -->
  <Target Name="FixLineDirectives" BeforeTargets="CoreCompile" DependsOnTargets="CallFsYacc;CallFsLex">
    <Exec Command="sed -i '' '/^# 0/d' Parser.fs" Condition="Exists('Parser.fs')" />
    <Exec Command="sed -i '' '/^# 0/d' Lexer.fs" Condition="Exists('Lexer.fs')" />
  </Target>

  <ItemGroup>
    <!-- Compile order matters in F#: Parser.fsi/.fs must precede Lexer.fs (Lexer opens Parser) -->
    <Compile Include="Parser.fsi" />
    <Compile Include="Parser.fs" />
    <Compile Include="Lexer.fs" />
    <Compile Include="Program.fs" />
  </ItemGroup>

  <ItemGroup>
    <PackageReference Include="FsLexYacc" Version="11.3.0" />
  </ItemGroup>
</Project>
```

**Key wiring facts (verified from FsLexYacc.targets and live build):**
- `<FsYacc Include="Parser.fsy"><OtherFlags>--module Parser</OtherFlags></FsYacc>` ‚Äî generates `Parser.fs` AND `Parser.fsi`.
- `<FsLex Include="Lexer.fsl"><OtherFlags>--unicode --module Lexer</OtherFlags></FsLex>` ‚Äî generates `Lexer.fs` (and `Lexer.fsi`, which is malformed and should NOT be included in `<Compile>`).
- `--module Parser` and `--module Lexer` flags are REQUIRED so generated files have the correct `module` header.
- The FsLexYacc MSBuild targets inject themselves automatically via the NuGet package ‚Äî no manual `<Import>` needed.
- **Critical compile ordering:** `Parser.fsi` and `Parser.fs` MUST appear before `Lexer.fs` because the generated `Lexer.fs` contains `open Parser`.

### A3. CRITICAL BUG: FsLexYacc 11.3.0 + .NET 10 Line Directive Incompatibility

**Verified by live probe.** This is a NEW finding not in prior STACK.md or PITFALLS.md.

`fsyacc` (and `fslex`) generate source line directives in this form:
```
# 0 ""
```
The F# 10 compiler (shipped with .NET 10.0.203) rejects this as a syntax error:
```
error FS0010: Unexpected integer literal. Expected identifier or other token.
```
(Line `# 0 ""` ‚Äî the integer `0` is unexpected in this position.)

**Effect:** `dotnet build` ALWAYS FAILS on the first try because this bad line is generated by fsyacc/fslex.

**Workaround (verified working):** Add a MSBuild `FixLineDirectives` target (shown in the .fsproj skeleton above) that runs `sed -i '' '/^# 0/d' Parser.fs` and similarly for `Lexer.fs` after generation but before compilation. The `DependsOnTargets="CallFsYacc;CallFsLex"` and `BeforeTargets="CoreCompile"` ordering is critical ‚Äî all three attributes are required.

**Implication for the agent run:** The agent will ALMOST CERTAINLY hit this build error on the first build attempt, regardless of how well it writes the grammar. This provides a GUARANTEED real error-and-fix cycle (build error ‚Üí agent reads error ‚Üí adds sed workaround or inline fix ‚Üí rebuild). However, the agent may struggle to diagnose it, so the planner should ensure the task 5 (build & test) prompt gives the agent enough context to iterate.

**Alternative workaround (simpler for the agent to discover):** Instead of the MSBuild target, the agent can manually edit `Parser.fs` to delete the `# 0 ""` line after the first failed build. Either approach works.

### A4. Canonical FsLex .fsl Content (VERIFIED WORKING)

```fsharp
{
open Parser
}

let digit = ['0'-'9']
let int = digit+
let white = [' ' '\t']+

rule tokenize = parse
| white    { tokenize lexbuf }
| int      { INT (int (System.String(lexbuf.Lexeme))) }
| '+'      { PLUS }
| '-'      { MINUS }
| '*'      { STAR }
| '/'      { SLASH }
| '('      { LPAREN }
| ')'      { RPAREN }
| eof      { EOF }
| _        { failwithf "Unexpected char: '%c'" (char lexbuf.Lexeme[0]) }
```

Key syntax details:
- `System.String(lexbuf.Lexeme)` converts the char array to string; `int` converts to int.
- The `{open Parser}` header block must come first; tokens are defined in Parser.fsy.
- `--unicode` flag in OtherFlags enables Unicode character range support.
- `--module Lexer` flag adds `module Lexer` header to generated file (required).

### A5. FsYacc .fsy Grammar ‚Äî With vs. Without Precedence

**CORRECT grammar (verified, gives 2+3*4=14, (2+3)*4=20, 10-3-2=5):**

```fsharp
%token <int> INT
%token PLUS MINUS STAR SLASH LPAREN RPAREN EOF

%left PLUS MINUS
%left STAR SLASH

%start start
%type <int> start

%%

start:
| expr EOF { $1 }

expr:
| expr PLUS expr  { $1 + $3 }
| expr MINUS expr { $1 - $3 }
| expr STAR expr  { $1 * $3 }
| expr SLASH expr { $1 / $3 }
| LPAREN expr RPAREN { $2 }
| INT { $1 }
```

**Precedence rule for FsYacc:** Operators declared on LATER `%left`/`%right` lines have HIGHER precedence. So `%left PLUS MINUS` (lower) then `%left STAR SLASH` (higher) is correct.

**NAIVE grammar (no %left declarations) ‚Äî bug characterization (verified by probe):**

Without `%left` declarations, fsyacc emits "16 shift/reduce conflicts" warnings and defaults to SHIFT on every conflict. The shift preference accidentally gives the correct precedence for `+` vs. `*` (because shifting `*` when we have `+` on the stack correctly delays the `+` reduction). BUT left-associativity of same-precedence operators is broken (right-associative by default).

Test results with naive grammar (no %left):
- `2+3*4` ‚Üí **14** (CORRECT ‚Äî shift preference happens to work for mixed-precedence case)
- `(2+3)*4` ‚Üí **20** (CORRECT ‚Äî parentheses override works correctly)
- `10-3-2` ‚Üí **9** (WRONG ‚Äî should be 5; right-associativity gives 10-(3-2)=9)
- `1-2+3` ‚Üí **-4** (WRONG ‚Äî should be 2; gives 1-(2+3)=-4)

**Critical implication for test case selection:** The two test cases in the CONTEXT (`2+3*4=14` and `(2+3)*4=20`) are INSUFFICIENT to detect the naive-grammar bug! Both pass even without `%left`. A THIRD test case is essential:

**Required test cases (all three):**
1. `2+3*4` ‚Üí `14` (detects wrong precedence order, e.g., `%left STAR SLASH` before `%left PLUS MINUS`)
2. `(2+3)*4` ‚Üí `20` (detects parenthesis/grouping issues)
3. `10-3-2` ‚Üí `5` (detects right-associativity bug from missing or partial `%left`)

Without test case #3, a naive grammar without any `%left` declarations will appear to pass.

**Other likely bugs from wrong %left:**
- If agent writes `%left STAR SLASH` THEN `%left PLUS MINUS` (wrong order): `2+3*4` ‚Üí 20 (caught by test 1).
- If agent writes `%left PLUS MINUS STAR SLASH` (all equal): `2+3*4` ‚Üí 20 (caught by test 1).
- If agent writes nothing (no %left): tests 1 and 2 pass but test 3 fails.

## B. OpenHands Headless Capture Mechanics

### B1. File Persistence and Shared Workdir (CONFIRMED)

**From 02-VERIFICATION-EVIDENCE.md and SDK source inspection:**

Each `openhands --headless -t "<task>"` invocation:
- Creates a **brand-new conversation** with a unique conversation ID.
- Has **zero memory** of any previous invocation ‚Äî no shared agent state, no conversation history, no tool call history.
- The agent's working directory is `OPENHANDS_WORK_DIR` ‚Äî the same directory for all invocations.
- **Files written by task N persist in the filesystem** and are visible to task N+1 (LocalWorkspace = host filesystem, no container isolation between invocations).

**Implication:** Each task prompt MUST be fully self-contained and describe the current state of the workdir. The agent cannot infer what previous tasks did from conversation history ‚Äî it can only `ls`, `cat`, and read existing files.

Good prompt pattern for task N+1:
> "Working directory: oh-workdir/calc/. The F# project skeleton exists (calc.fsproj, Lexer.fsl, Parser.fsy, Program.fs). Your task is: [specific task]. Constraints: [constraints]."

The agent's initial working directory is `OPENHANDS_WORK_DIR` (verified: working_dir shown in ping ObservationEvent = `/Users/ohama/projs/OpenHandsTests/oh-workdir`). If the project is in a subdirectory (`oh-workdir/calc/`), the agent must be told to `cd calc` or the prompt should describe relative paths.

### B4. Detecting Task Completion vs. Failure

**Normal completion** (use as "task succeeded" signal):
- JSONL contains `ActionEvent` with `action.kind == "FinishAction"` near the end, OR
- JSONL ends with `MessageEvent` from `source == "agent"` (final text-only response without tool call).
- CLI stdout shows: `"Agent finished"` followed by `"Goodbye! üëã"` and a conversation ID.
- CLI process exits with code **0** in both success AND failure cases.

**MaxIterations failure** (agent ran out of steps):
- JSONL contains `ConversationErrorEvent` with `code == "MaxIterationsReached"`.
- Default limit: **500 iterations per run** (no CLI flag to override; must change SDK config if needed ‚Äî almost certainly not needed for these small tasks).
- CLI exits with code **0**.

**Stuck/loop detection** (agent repeating same actions):
- NO `ConversationErrorEvent` is emitted to JSONL when stuck is detected.
- The run loop breaks after stuck pattern is detected (repeated action-observation pairs, 3+ text-only messages without tool use), and the app exits normally.
- Detect in JSONL by: scanning last 20 events for repeated commands (same `command` string appearing 3+ times consecutively).
- CLI stdout shows: `"Agent finished"` even when stuck (misleading).
- CLI exits with code **0**.

**Practical detection script** (for retry policy):
```bash
python3 -c "
import json, sys

logfile = sys.argv[1]
events = [json.loads(l) for l in open(logfile) if l.strip().startswith('{')]

# Check for explicit error
has_conv_error = any(e.get('kind') == 'ConversationErrorEvent' for e in events)
has_finish = any(e.get('action',{}).get('kind') == 'FinishAction' for e in events)

# Check for the expected outputs (14 for 2+3*4, 20 for (2+3)*4, 5 for 10-3-2)
all_text = ' '.join(
    item.get('text','')
    for e in events
    if e.get('kind') == 'ObservationEvent'
    for item in e.get('observation',{}).get('content',[])
)

passed_14 = '14' in all_text
passed_20 = '20' in all_text
passed_5_left_assoc = True  # can't reliably distinguish '5' from other context

print('ConversationError:', has_conv_error)
print('FinishAction:', has_finish)
print('Has 14 in output:', passed_14)
print('Has 20 in output:', passed_20)
print('Total events:', len(events))
" task5-buildtest.jsonl
```

### B5. Error-and-Fix Cycle ‚Äî Expected JSONL Signature

The error-and-fix cycle appears in the JSONL as:

1. **Build fails** (in task 5 or a follow-up task):
   - `TerminalAction`: `dotnet build 2>&1`
   - `TerminalObservation`: content contains the error, `exit_code != 0`
   - Possibly `TerminalObservation` containing `10-3-2` ‚Üí `9` (wrong answer)

2. **Agent diagnoses** (reads files, checks grammar):
   - Multiple `TerminalAction` events: `cat Parser.fsy`, `cat Parser.fs` (inspection)

3. **Agent fixes** (edits grammar or workaround):
   - `TerminalAction` events: writing corrected `Parser.fsy` (likely with `sed` or `tee`)

4. **Agent rebuilds and retests**:
   - `TerminalAction`: `dotnet build 2>&1`
   - `TerminalObservation`: content contains "Build succeeded"
   - `TerminalAction`: `dotnet run -- "2+3*4"`, `dotnet run -- "(2+3)*4"`, `dotnet run -- "10-3-2"`
   - `TerminalObservation`s: `14`, `20`, `5`

**To capture a clean error-and-fix in the JSONL:** The build-and-test task (task 5) prompt should test all three cases. If the build succeeds but test 3 (`10-3-2`) fails, the follow-up fix task can diagnose the grammar.

**Note on the line-directive bug:** The `# 0 ""` build error from FsLexYacc 11.3.0 is almost certain to appear on the FIRST build attempt. This is a REAL error that the agent must discover and fix. It constitutes a genuine error-and-fix cycle. The agent will likely try `dotnet build`, see the `FS0010` error on `Parser.fs:6`, then inspect `Parser.fs`, see the offending line, and remove it (or add the sed workaround). This satisfies RUN-03.

### B7. Retry/Adjust Policy ‚Äî Actionable Criteria

| Situation | Detected By | Action |
|-----------|-------------|--------|
| Task completed, tests pass | FinishAction + correct outputs in JSONL | Keep the log; move to next task |
| Task completed, tests fail (wrong output) | FinishAction + wrong output (e.g., 9 instead of 5) | Add a fix task with failing output shown explicitly |
| ConversationErrorEvent (MaxIterations) | `kind == "ConversationErrorEvent"` in JSONL | Retry task up to 2x; if still failing, split task into smaller pieces |
| Agent stuck (repeating actions) | No FinishAction, same commands repeated | Retry with a more explicit prompt, or split the task |
| Build never happened (no TerminalAction) | Zero TerminalAction events | Agent misunderstood task; rewrite prompt with clearer action verb |
| Build error `# 0 ""` (line directive) | `FS0010` in build output | This is the REAL error-and-fix; let agent handle it; if it can't in 2 tries, add the FixLineDirectives target to the scaffold prompt |
