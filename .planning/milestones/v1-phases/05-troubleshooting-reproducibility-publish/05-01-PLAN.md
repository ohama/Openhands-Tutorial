## 1. 환경 문제 (environment issues) — three subsections:

    `/Users/ohama/.colima/default/docker.sock`, `/var/run/docker.sock` absent.

    Fix: `colima start --cpu 4 --memory 8 --disk 60` then set
    `DOCKER_HOST=unix:///Users/ohama/.colima/default/docker.sock`. Note: Docker Desktop
    users do NOT need this; on Linux check `docker context inspect`.
  - **REAL-02 --override-with-envs REQUIRED**: without it, `LLM_MODEL`/`LLM_BASE_URL`/
    `LLM_API_KEY` are silently ignored. Symptom: agent starts but LLM calls fail (auth
    error / connection refused) with no message about ignored env vars. Fix: always pass
    `--override-with-envs`. Include the inline-env invocation snippet from §1.1 REAL-02.
    Tie in the `openai/` prefix note from §1.2 ("openai/ prefix APPLIES"): `LLM_MODEL`

  - **REAL-05 .NET 10 + FsLexYacc line-directive (FS0010) → FixLineDirectives**:
    fslex/fsyacc emit `# 0 ""` line directives; F# 10 (SDK 10.0.203) rejects them with
    FS0010. Fix: the `FixLineDirectives` MSBuild target in calc.fsproj strips them via
    `sed`. Include the verbatim XML `<Target Name="FixLineDirectives" ...>` block from
    §1.1 REAL-05. Note: macOS sed is `sed -i ''`, Linux is `sed -i`; the Pages workflow
    builds only the mdbook (ubuntu-latest), so this is a local-dev concern only.

## 2. 에이전트 동작 문제 (agent behavior issues) — two subsections:
  - **REAL-03 FsLex out-of-distribution for the 35B**: attempt 1 failed entirely on

    Fix: provide the lexer verbatim in the task prompt and let the agent focus on parser
    + evaluator (what attempt 2 did). Lesson: FsLex is genuinely poorly represented in
    LLM training data.
  - **REAL-04 file_editor security_risk validation error**: every file_editor/str_replace
    call by qwen-local fails with
    `Error validating tool 'file_editor': Failed to provide security_risk field in tool 'file_editor'.`
    The model's tool-call schema omits the required security_risk field. Fix: instruct in
    EVERY task prompt to create/edit files with bash only (printf/tee/quoted heredoc) and
    NOT use file_editor/str_replace. Include the verbatim IMPORTANT instruction block from
    §1.1 REAL-04. Result in attempt 2: zero file_editor errors.

## 3. 빌드 오류 (build errors):
  - **REAL-06 FsYacc %start/%type + LexBuffer.FromText**: the genuine error-and-fix cycle
    in task3 (4 failures → self-correct). Reproduce the table from §1.1 REAL-06:
    (1) FSY000 at least one %start declaration is required (wrote %type but no %start);
    (2) Parser.fsy(16,7): error parse error (used invalid `%start <int> start`);
    (3) same parse error; (4) FS0039 'LexBuffer<_>' does not define 'FromText'.
    Fix: separate `%start start` and `%type <int> start` onto their own lines, and use
    `LexBuffer<char>.FromString` instead of `LexBuffer.FromText`. Build then succeeded
    (`calc net10.0 성공 (0.7초)`). One sentence pointing readers to 4부 빌드와 테스트 단계
