## 더 크거나 다른 모델로 실험해 보기

이번 튜토리얼에서 가장 큰 장벽 중 하나는 FsLex 문법이 35B 모델의 학습 데이터 범위 밖에 있다는 점이었습니다. Attempt 1에서 세 번의 에이전트 호출과 137회의 TerminalAction이 모두 유효한 Lexer.fsl을 만들어내지 못했습니다.

더 큰 모델(70B, 또는 GPT-4 계열, Claude 계열 등 클라우드 모델)로 같은 작업을 실행하면 어떻게 다를지 비교해 보는 것은 좋은 다음 실험입니다. 모델 크기와 학습 데이터의 다양성이 이런 틈새 도구(niche tool)에서의 성공률에 어떤 영향을 미치는지 실제로 측정해볼 수 있습니다.

OpenHands 헤드리스 CLI의 `LLM_MODEL`과 `LLM_BASE_URL`만 바꾸면 같은 태스크 프롬프트를 다른 모델로 실행할 수 있습니다. 부록 A의 재현 가이드를 따라 직접 실행해 보세요.

## 다른 언어 또는 다른 도메인에서 에이전트 활용해 보기

F# FsLex/FsYacc 계산기는 의도적으로 틈새 기술 스택을 선택한 예제였습니다. 더 주류적인 언어 — Python, TypeScript, Go, Rust — 에서 비슷한 규모의 프로젝트를 에이전트에게 맡기면 어떻게 달라지는지 실험할 수 있습니다.

예를 들어:

- Python으로 JSON 파서 작성 (ply 또는 lark 라이브러리 사용)
- TypeScript로 간단한 CLI 도구 작성
- Go로 작은 HTTP 서버 작성

같은 5-태스크 분해 방식(scaffold → lexer/grammar → parser → evaluator → build-test)이 다른 언어에서도 효과적인지 비교해보면, 이 방법론이 얼마나 범용적인지 직접 확인할 수 있습니다.

## OpenHands 실험 범위 확장하기

이 튜토리얼은 헤드리스 CLI(`openhands --headless`)만 사용했습니다. OpenHands에는 이 외에도 탐색할 수 있는 영역이 있습니다:

- **더 긴 작업**: 단일 프롬프트로 전체 프로젝트를 만들게 하면 어떻게 되는가? 태스크를 더 작게 분해할수록 성공률이 어떻게 달라지는가?
- **기존 코드베이스 수정**: 새 프로젝트를 만드는 것이 아니라, 이미 존재하는 코드에 기능을 추가하거나 버그를 수정하는 작업은 어떻게 다른가?
- **다른 LLM 프록시**: litellm 외의 프록시(Ollama, LM Studio 등)와의 호환성은 어떻게 다른가?

각 실험은 tool calling, agent loop, memory 관리가 다른 조건에서 어떻게 동작하는지 이해하는 데 도움이 됩니다.

## 에이전트 실패 사례를 분석해 보기

이 튜토리얼에서 attempt 1의 실패가 오히려 가장 많은 것을 보여줬습니다. 에이전트가 왜 실패했는지 — FsLex 문법의 학습 데이터 부족, `%%` 구분자 혼동, file_editor 스키마 불일치 — 를 분석하는 과정이 에이전트 동작을 이해하는 실질적인 방법이었습니다.

앞으로 에이전트를 사용할 때, 실패가 발생하면 JSONL 로그를 직접 열어서 어느 이벤트에서 어떤 오류가 나왔는지 확인하는 습관을 들이면 좋습니다. 에이전트의 "생각 과정"은 JSONL 이벤트 시퀀스 안에 그대로 담겨 있습니다.
