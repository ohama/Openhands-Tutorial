# 태스크 계획 단계

OpenHands로 F# 계산기를 만들기 전에, 우리는 작업을 어떻게 나눌지 결정해야 했습니다. 에이전트에게 "F# 계산기를 만들어라"는 하나의 거대한 프롬프트를 주는 대신, 작업을 **다섯 개의 독립된 OpenHands 실행**으로 분해했습니다. 이 결정의 이유와 각 태스크의 실제 결과를 이 장에서 설명합니다.

---

## 다섯 개의 태스크로 분해

계산기 빌드는 다음 다섯 개의 OpenHands headless 실행으로 구성됩니다. 각 태스크는 독립된 `openhands --headless` 호출이며, 이전 태스크가 파일시스템에 남긴 결과물을 다음 태스크가 읽습니다.

| 태스크 | 역할 | 소요 시간 | TerminalActions |
|--------|------|-----------|----------------|
| task1-scaffold | 프로젝트 골격 생성 + calc.fsproj 복사 | 3분 6초 | 27회 |
| task2-lexer | Lexer.fsl 복사 | 16초 | 2회 |
| task3-parser | Parser.fsy 작성 + 빌드 오류 4회 자가 수정 (Program.fs도 작성) | 1분 17초 | 15회 |
| task4-evaluator | Program.fs CLI 연결 확인 및 정리 | 45초 | 14회 |
| task5-buildtest | 최종 빌드 + 3개 테스트 케이스 검증 | 32초 | 9회 |

**task1 (3분 6초)**은 27회의 TerminalAction이 대부분 파일 쓰기 방식의 탐색에 사용됐습니다. heredoc 방식, base64 인코딩, Python 스크립트 등 여러 접근을 시도하며 파일 쓰기에 어려움을 겪었습니다. 이것 자체가 에이전트 동작의 솔직한 기록입니다.

**task2 (16초, 2회)**는 Lexer.fsl을 단순히 복사하는 작업이었습니다 — 빠른 이유가 여기 있습니다.

**task3 (1분 17초)**이 이 튜토리얼의 핵심입니다. 에이전트가 Parser.fsy를 작성하고, 범위를 넘어 Program.fs도 작성하고, 4회의 빌드 실패를 스스로 수정했습니다. 이 과정은 '빌드와 테스트 단계' 장에서 자세히 다룹니다.

**task4 (45초)**는 "평가기(evaluator)"라는 이름과 달리, 별도의 평가 모듈을 만들지 않았습니다. task3에서 에이전트가 이미 Program.fs를 작성했기 때문에, task4는 파서 인터페이스 파일을 읽고 Program.fs를 약간 정리하는 데 그쳤습니다. 앞 장에서 설명한 대로, 산술 평가는 파서 문법 액션 안에서 이루어지므로 별도 평가기가 필요하지 않습니다.

**task5 (32초)**에서 빌드는 성공했고 세 가지 테스트 케이스 — `2+3*4=14`, `(2+3)*4=20`, `10-3-2=5` — 가 모두 올바른 값을 반환했습니다.

## 왜 렉서와 프로젝트 파일을 제공했는가

이 절은 이 튜토리얼에서 가장 솔직하게 기록해야 할 부분입니다.

`calc.fsproj`와 `Lexer.fsl` 두 파일은 태스크 프롬프트에 텍스트 그대로 포함되어 에이전트에게 전달됐습니다. 에이전트가 이 파일들을 직접 만든 것이 아닙니다.

**왜 이렇게 했는가?**

FsLex `.fsl` 파일 형식은 **Qwen2.5-35B 모델의 훈련 데이터에서 드문 형식**입니다. 시도 1에서 이것을 확인했습니다. 세 번의 별도 에이전트 실행(94회 + 27회 + 16회 = 137회 이상의 TerminalActions)이 모두 올바르지 않은 `.fsl` 파일을 생성했습니다. 구체적인 실패 패턴은 다음과 같습니다.

- FsYacc 문법의 `%%` 구분자를 FsLex 파일에도 삽입 (FsLex에는 `%%` 구분자가 없음)
- 잘못된 렉심(lexeme) 추출 패턴 사용
- 헤더 블록 들여쓰기 문제로 생성된 F# 파일이 컴파일 실패

세 번의 에이전트 실행이 모두 같은 종류의 오류를 반복했습니다. 이것은 속도의 문제가 아니라 **능력의 경계**입니다 — 이 모델은 FsLex `.fsl` 형식을 올바르게 생성하는 데 필요한 지식을 갖추고 있지 않습니다.

시도 2에서는 이 경계를 우회하기 위해 Lexer.fsl을 프롬프트에 직접 제공했습니다. `calc.fsproj`도 함께 제공한 이유는 `.NET 10 + FsLexYacc 11.3.0` 조합의 비직관적인 호환성 문제(`FixLineDirectives` 워크어라운드)가 있었기 때문입니다.

**에이전트의 실제 작업 범위:**

이 스캐폴딩 덕분에 에이전트의 진짜 기여를 명확하게 볼 수 있습니다.

- `Parser.fsy` — FsYacc 문법, 연산자 우선순위(`%left PLUS MINUS` / `%left STAR SLASH`), 산술 액션 작성
- `Program.fs` — F# CLI 진입점, `FSharp.Text.Lexing` API 연결
- **빌드 오류 자가 수정** — `%start` 누락 → 잘못된 `%start <int> start` 구문 → `LexBuffer<_>.FromText` 미존재 API 등 4회의 컴파일러 오류를 스스로 진단하고 수정

이 자가 수정 과정이 4부의 핵심입니다. '빌드와 테스트 단계' 장에서 각 오류와 수정 과정을 verbatim 오류 메시지와 함께 살펴봅니다.

---

> **개념 ↔ 행동: 계획 → 작성 → 테스트 → 실행 (Callout D)**
>
> 다섯 개의 태스크 분해는 1부에서 소개한 **plan → write → test → run** 방법론 — 즉 **탐색(Explore) → 분석(Analyze) → 구현(Implement) → 검증(Verify)** — 과 정확히 대응합니다.
>
> ```
> task1 (scaffold)   ←→  Explore   탐색: 환경 파악, 프로젝트 골격 생성
> task2 (lexer)      ←→  Analyze   분석: 필요한 입력 확인 및 준비
> task3 (parser)     ←→  Implement 구현: Parser.fsy 작성 + Program.fs 작성
> task4 (evaluator)  ←→  Implement 구현: Program.fs CLI 연결 정리
> task5 (buildtest)  ←→  Verify    검증: 최종 빌드 + 3개 테스트 케이스
> ```
>
> 그리고 task3 내부의 **4회 빌드 실패**는 Verify → Implement의 자가 수정 사이클 그 자체입니다. 에이전트는 `dotnet build` 결과(Verify)를 보고 Parser.fsy 또는 Program.fs를 수정(Implement)하는 반복을 4회 거쳐 빌드 성공에 도달했습니다.
>
> 1부 '계획 → 작성 → 테스트 → 실행' 절에서 "자가 수정은 별도로 설계된 기능이 아니라, agent loop와 observation 메커니즘의 자연스러운 결과"라고 설명했습니다. task3의 4회 사이클이 바로 그 자연스러운 결과의 실제 모습입니다.

## 다음 장으로

태스크 분해와 스캐폴딩 결정을 이해한 상태에서, '코드 작성 단계' 장에서는 task1(스캐폴딩)과 task3(파서 작성)의 실제 에이전트 행동 — 어떤 명령을 실행했는지, 어떤 파일을 썼는지 — 을 단계별로 살펴봅니다. '빌드와 테스트 단계' 장에서는 task3의 4회 빌드 오류와 자가 수정 사이클을 verbatim 오류 메시지와 함께 상세히 다룹니다.
