# 예제 프로젝트 소개

4부에서 OpenHands는 F# 계산기를 만듭니다. 이 장에서는 계산기가 무엇을 하는지, 어떤 파일로 구성되는지, 그리고 — 중요하게도 — 어떤 부분을 에이전트가 직접 만들었고 어떤 부분을 우리가 제공했는지를 미리 정리합니다.

F#을 처음 보는 독자도 걱정할 필요가 없습니다. 4부의 목적은 F#을 가르치는 것이 아닙니다. F# 언어 자체보다는 **에이전트가 어떻게 작동하는지**를 보는 것이 이 튜토리얼의 목표입니다. 코드가 등장하더라도 F# 문법을 외울 필요는 없습니다.

---

## 무엇을 만드는가

완성된 계산기는 정수 산술 표현식을 평가하는 **명령줄(command-line) 도구**입니다.

```
$ calc "2+3*4"
14

$ calc "(2+3)*4"
20

$ calc "10-3-2"
5
```

세 가지 결과가 모두 올바릅니다. 연산자 우선순위(STAR이 PLUS보다 먼저)와 왼쪽 결합(left-associativity: `10-3-2 = (10-3)-2 = 5`, `10-(3-2) = 9`가 아님)이 정확하게 처리됩니다.

이 계산기가 만들어지는 과정 — 에이전트가 파일을 쓰고, 빌드 오류를 읽고, 스스로 수정하는 모습 — 이 4부 전체의 내용입니다.

---

## tokenize → parse → evaluate 파이프라인

계산기 내부는 세 단계로 구성됩니다. F# 코드를 읽지 않아도 이 흐름을 이해할 수 있습니다.

```
입력 문자열: "2+3*4"
       |
       v
  [ tokenize (렉싱) ]
       |
       v
  토큰 스트림: INT(2) PLUS INT(3) STAR INT(4) EOF
       |
       v
   [ parse (파싱) ]
       |
       v
  구문 구조 + 평가 (우선순위·결합 반영)
       |
       v
  결과: 14
```

### 1단계 — tokenize (렉싱)

입력 문자열 `"2+3*4"`를 **토큰(token) 스트림**으로 분해합니다. 공백을 무시하고, 숫자를 INT 토큰으로, 연산자를 PLUS·MINUS·STAR·SLASH 토큰으로 변환합니다. 이 단계를 담당하는 것이 **렉서(lexer)**입니다.

예: `"2+3*4"` → `INT(2) PLUS INT(3) STAR INT(4) EOF`

### 2단계 — parse (파싱)

토큰 스트림을 **파서(parser)**가 받아서, 연산자 우선순위와 결합 방향에 따라 올바른 평가 순서를 결정합니다. `2+3*4`에서 `3*4`가 먼저 계산되어야 한다는 것은 파서 문법(grammar)에 선언된 `%left STAR SLASH`가 `%left PLUS MINUS`보다 나중에 선언됨으로써 처리됩니다 — 즉 STAR가 PLUS보다 높은 우선순위를 갖습니다.

### 3단계 — evaluate (평가)

중요한 점이 있습니다. **별도의 "평가기(evaluator)" 모듈은 없습니다.** 산술 계산은 파서 문법의 액션(action) 코드 안에서 즉시 이루어집니다. 예를 들어 `expr PLUS expr` 규칙의 액션은 `{ $1 + $3 }` — 즉 왼쪽 값과 오른쪽 값을 더합니다. 파싱이 곧 평가입니다.

`Program.fs`는 별도의 평가 로직이 없습니다. 렉서와 파서를 연결하고 명령줄 인자를 읽어 결과를 출력하는 **CLI 진입점(entry point)** 역할만 합니다.

## 네 개의 파일

계산기는 네 개의 파일로 구성됩니다.

```
calc/
├── Lexer.fsl      <-- 렉서 규칙 (FsLex DSL)
├── Parser.fsy     <-- 파서 문법 + 산술 액션 (FsYacc DSL)
├── Program.fs     <-- CLI 진입점 + lexer/parser 연결 (F#)
└── calc.fsproj    <-- 빌드 설정 (FsLexYacc 11.3.0)
```

각 파일의 역할과 **작성 주체**:

| 파일 | 역할 | 작성 주체 |
|------|------|-----------|
| `Lexer.fsl` | 토큰 분해 규칙 (FsLex DSL) | **제공됨 — 에이전트가 작성하지 않음** |
| `Parser.fsy` | 파서 문법 + 연산자 우선순위 + 산술 액션 (FsYacc DSL) | **에이전트 작성** |
| `Program.fs` | CLI 진입점 + 렉서/파서 연결 (F#) | **에이전트 작성** |
| `calc.fsproj` | 빌드 설정, FsLexYacc 패키지 참조, .NET 10 호환성 워크어라운드 | **제공됨 — 에이전트가 작성하지 않음** |

## 솔직한 공개: 스캐폴딩에 대하여

`Lexer.fsl`과 `calc.fsproj`는 에이전트가 만든 것이 아닙니다. 두 파일은 태스크 프롬프트에 **텍스트 그대로 포함되어** 에이전트에게 전달됐습니다. 에이전트는 이 내용을 받아 파일로 복사했을 뿐입니다.

에이전트가 **직접** 한 작업은 다음 세 가지입니다.

1. `Parser.fsy` — FsYacc 문법 파일 작성 (연산자 우선순위 선언 포함)
2. `Program.fs` — F# CLI 진입점 + FSharp.Text.Lexing API 연결
3. **빌드 오류 4회 자가 수정** — `%start` 선언 누락, 잘못된 `%start <int> start` 구문, `LexBuffer<_>.FromText` 미존재 API 등 컴파일러 오류를 스스로 진단하고 수정

왜 렉서를 제공했는지 — 그 이유는 다음 장 '태스크 계획 단계'에서 설명합니다. 간략히 말하면, FsLex `.fsl` 형식은 Qwen2.5-35B 모델의 훈련 데이터에서 드문 형식으로, 첫 번째 시도에서 에이전트가 올바른 `.fsl` 파일을 만들지 못했습니다. 이것은 이 튜토리얼에서 솔직하게 기록하는 **능력의 경계**입니다.

## 1부 개념과의 연결

4부 각 장에는 **개념 ↔ 행동 콜아웃**이 삽입됩니다. 에이전트가 `dotnet build`를 실행하는 장면에서 1부의 tool calling 개념을 다시 확인하고, 빌드 오류 4회 수정 사이클에서 agent loop와 자가 수정의 실제 모습을 봅니다. 태스크 분해 방식에서는 plan → write → test → run 방법론 — 즉 탐색(Explore) → 분석(Analyze) → 구현(Implement) → 검증(Verify) — 이 실제 작업에 어떻게 매핑되는지 확인할 수 있습니다.

이 콜아웃들은 1부에서 추상적으로 정의한 개념들이 실제 실행 기록 안에서 어떤 모습으로 나타나는지를 구체적으로 연결합니다.
