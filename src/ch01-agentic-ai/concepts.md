# 핵심 개념과 용어

에이전틱 AI를 설명할 때 반복적으로 등장하는 용어들이 있습니다. 이 장에서는 이후 모든 장의 토대가 되는 네 가지 핵심 개념을 정의합니다. 각 개념은 추상적인 이론이 아니라, 이 튜토리얼의 4부에서 OpenHands가 실제로 F# 계산기를 만드는 과정에서 직접 확인할 수 있는 것들입니다. 정의와 함께 그것이 어디에 등장하는지 미리 파악해 두면, 나중에 실제 실행 기록을 읽을 때 이해가 훨씬 빠릅니다.

개념들은 서로 독립적이지 않습니다. tool calling이 있어야 agent loop가 환경에 행동할 수 있고, agent loop가 있어야 plan → write → test → run 방법론이 단계별로 진행되며, 진행 과정 전체를 memory가 추적합니다. 이 네 가지를 함께 이해하면 에이전틱 AI의 동작 방식을 전체적으로 파악할 수 있습니다.

이 튜토리얼에서 사용하는 핵심 개념들은 OpenHands에만 국한된 것이 아닙니다. tool calling은 OpenAI, Anthropic, Gemini 등 주요 LLM API가 공통으로 지원하는 표준 메커니즘이고, agent loop와 memory 관리 방식도 대부분의 에이전틱 프레임워크가 공유하는 구조입니다. OpenHands는 이 보편적인 원리를 오픈소스로 구현한 것입니다.

---

## 툴 호출 (tool / function calling)

LLM은 텍스트를 생성하는 모델입니다. 그런데 에이전트 환경에서는 텍스트 생성 이상의 능력이 필요합니다 — 파일을 만들거나, 터미널 명령을 실행하거나, 코드를 빌드하는 것이 그 예입니다. **tool calling(툴 호출)**은 이 문제를 해결하는 메커니즘입니다.

동작 방식은 다음과 같습니다. 시스템이 LLM에게 사용 가능한 툴의 스키마(이름, 인자, 설명)를 미리 전달합니다. LLM은 일반 텍스트를 출력하는 대신, 구조화된 호출(이름 + 인자)을 출력해 환경에 행동을 지시합니다. 이 호출은 실제로 실행되고, 그 결과가 다시 LLM에게 전달됩니다. LLM은 이 결과를 보고 다음 행동을 결정합니다.

tool calling이 없다면 에이전트는 "이렇게 하세요"라는 글만 쓸 수 있습니다. tool calling이 있기 때문에 에이전트는 직접 "이것을 실행합니다"라고 선언하고 환경을 변화시킬 수 있습니다.

OpenHands에서 tool calling은 **ActionEvent**의 형태로 구체화됩니다. `CmdRunAction`은 셸 명령을 실행하고, `FileWriteAction`은 파일을 새로 만들고, `FileEditAction`은 기존 파일의 특정 부분을 수정합니다. `AgentThinkAction`은 부작용 없이 순수하게 추론만 수행하는 특수한 툴입니다. 이것들은 모두 LLM이 "이 툴을 이 인자로 호출하라"고 결정한 결과입니다.

tool calling은 에이전트와 반응형 챗봇을 구분하는 가장 근본적인 기술적 차이입니다. 챗봇은 텍스트로만 응답하지만, 에이전트는 tool calling을 통해 실제 세계를 변화시킵니다.

4부 '코드 작성 단계'에서 OpenHands가 FileWriteAction과 FileEditAction으로 `.fsl`·`.fsy` 파일을 만드는 장면에서 tool calling을 직접 볼 수 있습니다.

## 에이전트 루프 (agent loop)

에이전트가 목표를 받으면, 그 목표를 한 번에 달성하는 것이 아니라 **루프**를 반복하며 점진적으로 나아갑니다. 이 반복 구조를 **agent loop(에이전트 루프)**라고 합니다.

루프는 크게 다음 단계로 구성됩니다.

1. **프롬프트 준비** — 지금까지의 이벤트(메시지, 행동, 관찰)를 LLM 입력 형식으로 변환한다.
2. **LLM 호출** — LLM이 다음 행동을 결정한다(tool call 또는 메시지).
3. **행동 분류 및 실행** — 결정된 행동을 환경에서 실행한다(파일 편집, 명령 실행 등).
4. **관찰 추가** — 행동의 결과(stdout, stderr, diff, 파일 내용 등)를 이벤트 기록에 추가한다.
5. **반복 또는 종료** — 목표가 달성되지 않았으면 1번으로 돌아간다.

핵심은 **루프가 자율적**이라는 점입니다. 사람이 "이제 다음 단계로 가라"고 지시하지 않아도, 에이전트가 관찰 결과를 보고 스스로 다음 행동을 결정합니다. 빌드가 실패하면 오류 메시지를 읽고 수정 코드를 작성하고, 성공하면 테스트를 실행하는 식입니다.

OpenHands에서 이 루프는 `Conversation.step()`이라는 함수가 담당하며, `while not finished:` 구조 안에서 반복됩니다. 루프는 `AgentFinishAction`이 발생하거나, 반복 횟수·비용 한도에 도달하거나, 복구 불가능한 오류가 생길 때 종료됩니다. 오류가 발생해도 오류 메시지 자체가 다음 루프의 입력이 되기 때문에, 많은 경우 에이전트가 스스로 복구합니다.

4부 전체(계획 → 코드 작성 → 빌드와 테스트)가 이 agent loop가 한 바퀴씩 도는 모습입니다. 각 장은 루프의 한 반복이 아니라, 여러 번의 반복이 특정 목적(계획 수립, 코드 생성, 빌드 검증)에 집중되는 단계를 보여줍니다.

## 계획 → 작성 → 테스트 → 실행 (plan → write → test → run)

에이전트가 복잡한 작업을 수행할 때, 곧바로 코드를 작성하지 않습니다. OpenHands의 핵심 에이전트인 **CodeActAgent**는 작업을 네 가지 단계로 나눠 수행하는 방법론을 따릅니다.

1. **탐색(Explore)** — 현재 환경을 파악한다. 어떤 파일이 있는지, 어떤 도구를 쓸 수 있는지 확인한다. `CmdRunAction`으로 `find`, `grep`, `cat` 같은 명령을 실행한다.
2. **분석(Analyze)** — 탐색 결과를 바탕으로 가설을 세운다. 어떤 접근 방식이 올바른지, 어떤 파일을 만들어야 하는지 추론한다. `AgentThinkAction`으로 순수 추론을 수행한다(부작용 없음).
3. **구현(Implement)** — 최소한의 변경으로 목표를 달성한다. 전체 파일 재작성보다 `FileEditAction`(문자열 교체 편집)을 우선한다.
4. **검증(Verify)** — 테스트를 실행해 목표가 달성됐는지 확인한다. `CmdRunAction`으로 빌드·실행 명령을 수행하고 출력을 관찰한다. 실패하면 다시 분석 단계로 돌아간다.

이 방법론은 사람이 처음 보는 코드베이스에 접근하는 방식과 유사합니다. 먼저 파악하고, 계획을 세우고, 조심스럽게 변경하고, 검증합니다. 에이전트가 무작위로 파일을 수정하거나 전체를 지우고 다시 쓰지 않는 이유가 이 구조에 있습니다.

특히 **검증 단계의 자가 수정(self-correction)**이 중요합니다. 빌드 오류는 `CmdOutputObservation`의 stderr로 EventLog에 기록됩니다. 이 오류 메시지가 다음 루프 반복의 프롬프트에 포함되어 LLM이 그것을 읽고 수정 행동을 선택합니다. 자가 수정은 별도로 설계된 기능이 아니라, agent loop와 observation 메커니즘의 자연스러운 결과입니다.

4부의 각 장이 이 단계 하나씩에 대응합니다. '태스크 계획 단계'는 Explore와 Analyze를 보여주고, '코드 작성 단계'는 Implement를, '빌드와 테스트 단계'는 Verify와 자가 수정 사이클을 보여줍니다.

## 메모리와 컨텍스트 (memory / context window)

에이전트 루프가 반복될수록 해야 할 일이 쌓입니다. 어떤 파일을 만들었는지, 어떤 명령이 실패했는지, 어떤 오류 메시지가 나왔는지 — 이 모든 것을 에이전트가 기억해야 다음 단계를 올바르게 결정할 수 있습니다. 이것이 **memory(메모리)**와 **context window(컨텍스트 창)**의 역할입니다.

OpenHands는 모든 상호작용을 **EventLog**에 순서대로 기록합니다. EventLog는 append-only(추가만 가능한) 구조입니다 — 한번 기록된 이벤트는 삭제되지 않습니다. 시스템 프롬프트, 사용자 메시지, 에이전트의 tool call, 툴 실행 결과, 오류 메시지, 자가 수정 코드 모두 여기에 쌓입니다. 각 이벤트는 고유 ID, 타임스탬프, 출처(사용자·에이전트·환경)를 가집니다.

LLM에는 한 번에 처리할 수 있는 입력의 최대 길이(context window)가 있습니다. EventLog가 길어지면 전부 LLM에 전달할 수 없게 됩니다. OpenHands는 이 문제를 **condensation(압축)**으로 해결합니다. 기록이 일정 길이(기본값: 80개 이벤트)를 넘으면, 중간 이벤트들을 LLM이 요약해 **CondensationSummaryEvent**로 압축합니다. 원본 이벤트는 EventLog에 남아 재현(replay)이 가능하지만, LLM에는 압축된 요약본이 전달됩니다. 이 방식으로 전체 기억을 유지하면서도 context window 한도를 넘지 않습니다.

이 설계의 중요한 함의는, 에이전트가 오래 전에 발생한 오류나 결정을 "잊지 않는다"는 것입니다. 30번째 반복에서 발생한 빌드 오류가 50번째 반복의 수정 결정에 영향을 줄 수 있습니다. 압축 이후라도 요약 형태로 해당 맥락이 유지되기 때문입니다.

4부의 긴 다단계 실행에서 EventLog가 쌓이고 필요하면 context가 압축되는 상황을 볼 수 있습니다. F# 계산기 작업처럼 10회 이상의 tool call이 이어지는 경우, CondensationSummaryEvent가 작동해 전체 기억이 유지되면서도 context window 한도를 넘지 않는 방식이 실제 실행 기록에서 어떻게 보이는지 확인할 수 있습니다.

## 개념 ↔ OpenHands 컴포넌트 ↔ 4부 등장 위치

이 표는 위에서 정의한 개념들이 OpenHands V1의 어떤 컴포넌트로 구현되는지, 그리고 4부의 어느 장에서 실제로 볼 수 있는지를 요약합니다. 2부 아키텍처 장에서 각 컴포넌트를 자세히 다룰 때 이 표가 다시 참조됩니다.

| 개념 | OpenHands V1 컴포넌트 | 4부에서 보는 곳 |
|------|----------------------|----------------|
| Tool / function calling | `ActionEvent` 서브타입 (CmdRunAction, FileEditAction 등) + LLM에 전달되는 툴 스키마 | 코드 작성 단계 (FileWriteAction, FileEditAction으로 .fsl/.fsy 생성) |
| Agent loop | `Conversation.step()` (`while not finished:` 루프) | 태스크 계획, 코드 작성, 빌드와 테스트 — 모든 단계 |
| Plan → Write → Test → Run | CodeActAgent의 4단계: Explore → Analyze → Implement → Verify | 각 장이 한 단계에 대응 |
| Memory / context window | `EventLog` (append-only 기록); `CondensationSummaryEvent` (압축) | 긴 다단계 실행 전체; 특히 빌드와 테스트 단계에서 오류-수정 맥락 유지 |

지금은 이 개념들과 컴포넌트 이름의 연결만 기억해 두면 충분합니다. 2부에서 각 컴포넌트의 내부 동작을 자세히 살펴보고, 4부에서 실제 실행 기록을 통해 이 개념들이 어떻게 작동하는지 눈으로 확인하게 됩니다.

## 이 장의 핵심 정리

네 가지 개념을 한 문장씩 요약하면 다음과 같습니다.

- **Tool calling** — LLM이 텍스트 대신 구조화된 툴 호출(이름 + 인자)을 출력해 환경에 직접 행동을 지시하는 메커니즘; OpenHands에서는 ActionEvent 서브타입으로 구현됩니다.
- **Agent loop** — 목표가 달성될 때까지 프롬프트 준비 → LLM 호출 → 행동 실행 → 관찰 기록을 반복하는 while-not-finished 구조.
- **Plan → Write → Test → Run** — 에이전트가 복잡한 작업을 탐색·분석·구현·검증의 네 단계로 나눠 체계적으로 수행하는 방법론.
- **Memory / context window** — 모든 이벤트를 EventLog에 기록해 진행 상황을 추적하고, context가 길어지면 CondensationSummaryEvent로 압축해 LLM 입력 한도를 관리하는 방식.

이 개념들을 이해한 상태로 2부 OpenHands 아키텍처를 읽으면, 각 컴포넌트가 왜 그 방식으로 설계됐는지 자연스럽게 이해됩니다. 그리고 4부에서 실제 실행 기록을 읽을 때, 화면에 나타나는 각 이벤트가 이 개념들 중 어디에 해당하는지 바로 파악할 수 있게 됩니다.

개념의 이해가 튜토리얼 전체를 이해하는 토대입니다. 다음 장부터 OpenHands의 구체적인 구조로 들어가겠습니다.
